# 项目文档写作规范

## 目标

本规范用于约束 OUC xv6 Lab Kit 的项目文档、教学文档、复现文档和提交文档。所有 `docs/` 下的 Markdown 文件都应服务于一个清晰目标：让学生能按步骤学习，让教师能布置与验收，让队友能独立复现，让评审能判断项目事实、贡献边界和证据可靠性。

## 适用对象

本规范适用于维护 `docs/`、`docs/final/`、`labs/` 中引用文档、提交材料说明和阶段审查记录的成员。它也适用于后续把本项目改造成课程作业、实验讲义或比赛提交包的维护者。

## 内容范围

文档必须覆盖目标、读者、前置条件、操作步骤、验证方式、常见失败、证据位置和边界条件。涉及实现事实时，应指向 `patches/`、`scripts/`、`labs/` 或 `submissions/evidence_manifest.md`；涉及历史状态时，应明确它是 historical record，不能让读者误以为仍是 current final。

## 结构规范

每份正式文档建议采用“目标、适用对象、内容范围、操作或设计说明、质量标准、边界条件”的结构。教学文档可以增加“课堂使用方式”和“学生产出”；复现文档应增加“命令、预期输出、失败处理”；审查文档应增加“结论、证据、风险、后续动作”。

## 语言风格

文档使用中文技术写作风格，句子应具体、可验证、少套话。描述项目价值时用事实支撑，不使用空泛形容；描述验证结果时写清命令、commit、suite 和证据位置；描述限制时直接给出边界，不把观察型 syscall 写成完整子系统实现。

## 质量标准

合格文档应满足五项要求：事实可追溯，命令可复现，读者路径清晰，历史与当前状态分离，边界不夸大。正式文档还应能被 `scripts/check-docs-consistency.sh` 和 `scripts/check-final-hygiene.sh` 这类门禁脚本检查。

## 边界条件

文档不能伪造队友复现、日志、视频或 SHA256；不能把 timeout 捕获写成长期稳定性测试；不能把 `pgcount()` 写成完整内存管理；不能把 `fcount()`、`fdcount()`、`fdinfo()` 写成完整文件系统；不能把 Lab5 写成新增内核机制。
