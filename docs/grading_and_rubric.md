# 评分建议与扣分点

> 维护时间：2026-06-10（stage13 增补批量解析说明；rubric 自 stage12）。配合各 lab 的 `student_tasks.md` 使用——每份任务书自带 100 分细则，这里给跨 lab 的统一原则和总评建议。

批量收 summary 时先过一遍 `bash scripts/grade-summaries.sh <目录>`：它把"overall 与单项矛盾、缺新测试项、内容雷同、commit 不符"的文件标出来，帮你决定优先抽查谁。**它不自动打分**，只帮助发现风险；干净 PASS 不代表满分，失败但定位清楚也应获得过程分。下面的维度和扣分表才是评分依据。

## 四个评分维度

| 维度 | 看什么 | 占比建议 |
| --- | --- | ---: |
| 功能正确性 | patch 能从声明的基底 apply、`make` 通过、自检命令真实通过 | 35% |
| 代码理解 | 能讲清调用路径、锁的归属、copyout 的检查点；答辩/抽查不靠背 | 25% |
| 实验报告 | 命令与输出真实成对出现、预测-验证结构完整、他人可照着重跑 | 25% |
| 复现证据 | summary 块完整、commit/时间自洽、失败记录真实 | 15% |

## 各 Lab 重心

- **Lab1**：七件套齐不齐、破坏-修复实验的症状解释。理解题重点：stub 缺失和 dispatch 表缺失的症状差异。
- **Lab2**：锁的逐路径分析是大头；负向用例（`pcount(99) = -1`）有没有。
- **Lab3**：预测-验证有没有"事前预测"；`walk(..., 0)` 第三个参数；T4 加字段后的 padding 分析。
- **Lab4**：`dup` 对 fdcount/fcount 影响的解释；`fdinfo` 三条校验各拦什么；不固定值纪律。
- **Lab5**：summary 真实性、故障记录、证据边界三句话。机制分占比降低，工程素养占比升高。

## 统一扣分表

| 行为 | 处理 |
| --- | --- |
| 测试程序打印写死的"正确输出" | 该任务 0 分，按伪造谈话 |
| summary 块手工篡改（commit/时间对不上） | 复现证据维度 0 分 |
| 把单次观察值写成必然值（"pcount 恒为 1"） | 报告维度扣 5-10 分 |
| 把观察实验写成"实现了内存管理/文件系统" | 理解维度扣 5-10 分（概念错误） |
| 提交包含 `external/xv6-riscv/` 全目录或 `logs/*.log` | 报告维度扣 5 分，退回重交 |
| 报告无任何失败/异常记录 | 复现维度扣 5 分，并列为抽查对象 |
| patch 无法从声明的基底 apply | 功能维度减半，给一次修复机会 |
| 用别人/历史的 PASS 充当自己的证据 | 复现证据维度 0 分 |

## 加分项（每项 +2~5，封顶 10）

- 选做挑战完成且解释正确。
- 思考题答出超出讲义的内容（如自己读了 `scheduler()`/`filedup()` 源码并给出行号）。
- 发现并如实报告了讲义/脚本的真实问题（修不修都行，报告准确就加）。

## 等第参考

- **优（90+）**：必做全对 + 至少一个选做 + 抽查对答如流。
- **良（80-89）**：必做全对，理解题有小瑕疵。
- **中（70-79）**：功能通过，但报告可重跑性差或理解靠背。
- **及格（60-69）**：主线命令能跑通，理解和报告均薄弱。
- **不及格**：apply/make 不过且无定位过程，或出现伪造。
