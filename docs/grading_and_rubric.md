# 评分建议与扣分点

## 目标

本文提供跨 lab 的统一评分原则，帮助教师和助教在功能、理解、报告和复现证据之间取得平衡。

## 适用对象

适用于教师、助教和课程维护者。每个 lab 的具体任务仍以 `student_tasks.md` 为准。

## 评分维度

功能正确性建议占 35%，重点看 patch 能否从声明基底应用、make 是否通过、验证命令是否真实通过。代码理解建议占 25%，重点看 syscall 路径、锁、copyout、页表和 fd/file 关系。实验报告建议占 25%，重点看命令与输出是否成对出现、预测-验证是否完整。复现证据建议占 15%，重点看 summary、suite、时间和失败记录是否自洽。

## 统一扣分

测试程序打印写死结果，相关任务记 0 分并按伪造处理。summary 手工篡改，复现证据维度记 0 分。把 `pcount`、`fcount` 或子进程状态写成固定值，按概念错误扣分。把观察实验写成完整内存管理或完整文件系统，按理解错误扣分。提交 `external/xv6-riscv/` 或 raw logs，应退回整改。

## 加分方向

可给完成选做挑战、解释源码行号、发现真实文档/脚本问题并准确报告的学生加分。加分不应掩盖主线复现失败。

## 质量标准

评分应同时看机器结果和人工理解。干净 PASS 不代表满分，失败但定位清楚也应获得过程分。抽查时优先选择 summary 异常、缺新测试项、验证身份不符或内容高度雷同的提交。

## 边界条件

评分脚本不自动打分，只帮助发现风险。最终成绩由教师按课程目标、任务书和人工抽查确定。

## 内容范围

本文内容限定在当前标题所对应的项目记录、教学说明、复现步骤或审查结论内。涉及 当前正式验证范围、历史证据、验证命令和证据材料时，应以 `submissions/evidence_manifest.md`、`patches/integrated-labs/README.md` 和相关脚本为准。

## 结构规范

文档应按“背景或问题、过程或设计、证据或命令、风险和后续动作”的顺序组织。历史文档可保留阶段性记录，但必须避免覆盖 当前正式验证范围。

## 语言风格

使用中文技术写作风格，命令、文件名、syscall 名和 SHA256 保持原样。结论应有证据支撑，不使用宣传性、绝对化或无法验证的表述。
