## pcount 实现与锁分析

`sys_pcount()` 使用 `argint(0, &state)` 从用户态读取状态编号。状态范围检查使用 xv6 内核中的 `UNUSED` 到 `ZOMBIE` 枚举范围：

```c
if (state < UNUSED || state > ZOMBIE)
  return -1;
```

统计过程遍历 `proc[]`，并在读取每个 `p->state` 时持有对应 `p->lock`：

```c
for (p = proc; p < &proc[NPROC]; p++) {
  acquire(&p->lock);
  if (p->state == state)
    count++;
  release(&p->lock);
}
```

教学意义：

- 让学生看到 `proc[]` 不只是单 pid 查找，也可以用于全表统计。
- 强调 `p->state` 是会被调度、sleep/wakeup、exit/wait 等路径修改的共享状态，读取时需要考虑锁。
- 引出“快照”语义：返回值只是 syscall 执行时刻的观察结果，打印时状态可能已变化。

## pchildtest 子进程观察设计

`pchildtest` 的目标是避免 `pstate(self)` 几乎总是 `RUNNING` 的教学局限：

1. 父进程调用 `fork()` 创建子进程。
2. 子进程执行 `pause(50)`，制造可被观察的时间窗口。
3. 父进程短暂 `pause(2)` 后调用 `pstate(child_pid)`。
4. 父进程打印 `pstate(child) = <state> (<STATE_NAME>)`。
5. 父进程同时打印 `pstate(-1) = -1` 作为负向观察。
6. 父进程 `kill(child)` 并 `wait()` 回收。

边界说明：子进程状态受调度时序影响，本地日志中观察到过 `RUNNABLE` 和 `SLEEPING`。自动验证只匹配 `pstate(child) =` 前缀，不固定要求某个状态。

## 命名问题与真实修正

用户请求中建议用户程序名为 `pstatechildtest`。第一次实现后，构建/boot 过程中真实触发 `mkfs` 失败，原因是 xv6 文件系统目录项名称长度受 `DIRSIZ` 限制，`_pstatechildtest` 过长。

处理方式：

- 不伪造 `pstatechildtest` 运行成功。
- 将实际用户程序缩短为 `pchildtest`。
- 保持输出文本为 `pstate(child) = ...`，方便教学和自动验证。
- 在文档中记录该命名限制，作为 xv6 文件系统限制的附带教学点。

## 教学价值

相较 lab2 v0.1，v0.2 有三个明确提升：

- 从“观察自己几乎总是 RUNNING”扩展为“统计进程表中的某类状态数量”。
- 从单进程观察扩展到子进程观察，引出调度时序不确定性。
- 增加负向测试，让学生看到无效状态输入和无效 pid 的返回约定。

该实验可以自然串联以下概念：

- `struct proc`
- `enum procstate`
- `proc[]`
- `p->lock`
- syscall 参数传递
- 子进程与调度时序
- 内核观察结果的快照语义
- 用户态和内核态对状态编号的语义耦合
