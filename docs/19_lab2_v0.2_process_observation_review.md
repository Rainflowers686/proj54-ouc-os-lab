# lab2 v0.2 process observation 复现审查

本文档记录 lab2 进程状态观察实验 v0.2 的真实扩展与复现结果。该扩展进入 `patches/integrated-labs/`，作为综合演示序列的 `0004`，不替换 lab1/lab2 的 independent patch。

- stage5a：完成 `0004` patch（`pcount` / `pcounttest` / `pchildtest`）并首次本地验证。
- stage5b：严格红队复审 + 从 clean baseline 独立重新复现。结论：**`0004` patch 未做任何代码改动**，本轮只补全文档证据与诚实边界说明。

## 总体结论

lab2 v0.2 已完成并通过 stage5b 本地重新验证（2026-06-04，WSL2 Ubuntu-24.04）：

- 新增 `pcount(int state)` syscall，编号为 `SYS_pcount = 25`。
- 新增用户程序 `pcounttest`，覆盖 `pcount(RUNNING) =` 和 `pcount(99) = -1`。
- 新增用户程序 `pchildtest`，通过子进程观察 `pstate(child) = ...`。
- clean baseline + integrated `0001-0004` 可顺序应用。
- `make` 在 `-Wall -Werror` 下成功，`kernel/sysproc.o` 无警告编译。
- boot evidence、hello、add2test、pstatetest、pcounttest、pchildtest 均捕获到真实关键输出。

红队判断：v0.2 相比 v0.1 **确有教学深度提升**（详见“教学价值”），`0004` patch 干净、最小、可复现、无安全问题。该结论只代表 timeout 自动捕获到关键文本，**不代表**长期稳定性测试、人工交互录屏或第二名队员独立复现。

## pcount 实现与锁分析

`sys_pcount()` 完整实现（来自 `0004` patch，应用到 `kernel/sysproc.c`）：

```c
uint64
sys_pcount(void)
{
  int state;
  struct proc *p;
  int count = 0;

  argint(0, &state);
  if (state < UNUSED || state > ZOMBIE)
    return -1;

  for (p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if (p->state == state)
      count++;
    release(&p->lock);
  }

  return count;
}
```

### 参数读取

`argint(0, &state)` 从用户态读取第一个整数参数。本 baseline 中 `argint` 的签名是 `void argint(int n, int *ip)`（`*ip = argraw(n);`），**没有返回值**，因此不存在“忽略返回值”的问题；这与 baseline 自带的 `sys_pause`、以及本项目的 `sys_add2`、`sys_pstate` 的调用方式一致。对寄存器参数而言 `argint` 不会失败。

### 状态范围检查

baseline 的状态枚举为：

```c
enum procstate { UNUSED, USED, SLEEPING, RUNNABLE, RUNNING, ZOMBIE };
```

即合法范围是 `0..5`。检查 `if (state < UNUSED || state > ZOMBIE) return -1;` 等价于 `state < 0 || state > 5`，可同时挡住负数和过大值。`state` 是 `int`、`UNUSED/ZOMBIE` 是枚举常量，二者比较在 `-Wall -Werror` 下编译无警告（stage5b 已实测 `kernel/sysproc.o` 干净编译）。

### 锁与并发

统计过程遍历 `proc[]`（`NPROC = 64`），在读取每个 `p->state` 时持有对应 `p->lock`：

```c
for (p = proc; p < &proc[NPROC]; p++) {
  acquire(&p->lock);
  if (p->state == state)
    count++;
  release(&p->lock);
}
```

红队确认的三个要点：

1. **不会自死锁**：执行 syscall 的调用进程在内核态运行时并不持有自己的 `p->lock`（该锁只在调度切换路径上短暂持有）。因此当循环走到调用者自己的 `proc` 槽位时，`acquire(&p->lock)` 是安全的，不会因为 xv6 自旋锁不可重入而死锁。
2. **逐槽加锁、非整表快照**：本实现一次只持有一个 `p->lock`，读完即释放。它**不是**对整张进程表的原子快照——在释放 `proc[i].lock` 与获取 `proc[i+1].lock` 之间，进程状态可能被调度/`wakeup`/`exit` 改变。这是 xv6 中**唯一正确**的做法（不应同时持有多个 `p->lock`，也没有可用的全局进程表锁），代价是返回值是一个“近似聚合”，而非某一瞬间的精确切片。这一点应作为诚实的局限而非缺陷来理解。
3. **返回值的快照语义**：返回的数字只是 syscall 执行期间的观察结果，用户态打印时实际状态可能已变化。

### 为什么 `pcount(RUNNING)` 实测是 1（且不固定承诺）

调用 `pcount(RUNNING)` 时，**调用进程自身正处于 RUNNING**（它正在执行该 syscall），所以结果按构造 `>= 1`。QEMU 默认 `CPUS = 3`，理论上最多可有 3 个进程同时 RUNNING；但在 `pcounttest` 发起调用的瞬间，其它 hart 通常停在调度器空转（没有进程 → 不计入）。stage5b 实测稳定得到 `pcount(RUNNING) = 1`（仅调用者自己）。文档据此承诺输出**形如** `pcount(RUNNING) = <n>`，自动验证只匹配前缀 `pcount(RUNNING) =`，**不**把数值钉死为 1。

## pcount(99) 负向测试价值

`pcounttest` 故意调用一次越界状态 `pcount(99)`：

- `99 > ZOMBIE(5)`，命中范围检查，内核返回 `-1`。
- stage5b 实测输出 `pcount(99) = -1`。

教学价值：让学生看到 syscall 的**输入校验**与**错误返回约定**是实现的一部分，而不是只测“happy path”。它与 `pstate(-1) = -1`（见 `pchildtest`）一起构成两个真实的负向用例，证明无效状态号和无效 pid 都被正确拒绝。

## pchildtest 子进程观察设计

`pchildtest` 的目标是避免 `pstate(self)` 几乎总是 `RUNNING` 的教学局限：

1. 父进程调用 `fork()` 创建子进程。
2. 子进程执行 `pause(50)`，制造可被观察的时间窗口。
3. 父进程短暂 `pause(2)` 后调用 `pstate(child_pid)`。
4. 父进程打印 `pstate(child) = <state> (<STATE_NAME>)`。
5. 父进程同时打印 `pstate(-1) = -1` 作为负向观察。
6. 父进程 `kill(child)` 并 `wait()` 回收，避免僵尸或挂起。

> 说明：本 baseline 把传统 xv6 的 `sleep` 系统调用改名为 `pause`（`sys_pause` 实为按 tick 休眠，见 baseline `kernel/sysproc.c`）。因此 `pause(50)` = 休眠 50 个 tick。详见“baseline 依赖说明”。

### 调度时序不确定性（stage5b 实测证据）

子进程状态受调度时序影响。stage5b 在**同一次 boot** 内由脚本反复发送 `pchildtest`（约 28 次），日志中**同时**出现：

```
pstate(child) = 3 (RUNNABLE)
pstate(child) = 2 (SLEEPING)
```

即：同一个二进制、同一次启动，仅因调度时序不同，父进程观察到的子进程状态在 `RUNNABLE` 与 `SLEEPING` 之间变化。这是“调度不确定性”最直接的真实证据。自动验证只匹配 `pstate(child) =` 前缀，**不固定要求**某个状态。

## 命名问题与真实修正

用户最初建议用户程序名为 `pstatechildtest`。第一次实现时，构建 fs image 阶段真实触发 `mkfs` 失败，原因是 xv6 文件系统目录项名称长度受 `DIRSIZ` 限制，`_pstatechildtest` 过长。

处理方式：

- 不伪造 `pstatechildtest` 运行成功。
- 将实际用户程序缩短为 `pchildtest`。
- 保持输出文本为 `pstate(child) = ...`，方便教学和自动验证。
- 在文档中记录该命名限制，作为 xv6 文件系统限制的附带教学点。

红队核查：全仓库 tracked 文件中所有 `pstatechildtest` 字样都是“解释改名原因”的历史说明，**没有**任何地方把它当作实际命令名来调用。命名已全仓一致为 `pchildtest`。

## usys.pl 文件模式警告（stage5b 实测，已解释）

stage5b 从 clean baseline 应用 `0001-0004` 时，每个 patch 的 `git apply` 都打印：

```
warning: user/usys.pl has type 100644, expected 100755
```

红队判定：

- **这是无害警告，patch 仍然成功应用**（`[OK] patch applied` 对每个 patch 都出现，`make` 成功）。
- 成因：patch 头记录 `user/usys.pl` 的 index 模式为 `100755`（可执行，符合上游 xv6 原状），但工作树位于 Windows NTFS 经 WSL 挂载的 `/mnt/d`，该挂载点 `core.filemode` 通常为 `false`，checkout 出来的文件模式是 `100644`，于是 git 报告模式不一致。
- **环境相关**：在原生 ext4 检出（保留可执行位）上不会出现该警告。
- **不做 patch 改造**：不为了消除该警告而重导 `0001-0004`（那会牵动 lab1/lab2 三个已提交 patch 并需要整轮重验，风险大于收益）。上游 `usys.pl` 本就是 `100755`，警告的“错”在于挂载点不保留模式位，而非 patch 本身。
- 处置 = **文档解释**（即本节），不规范化 patch。

## 教学价值

相较 lab2 v0.1（仅 `pstate(self)` ≈ 总是 RUNNING），v0.2 有三个明确提升：

- 从“观察自己几乎总是 RUNNING”扩展为“统计进程表中某类状态的数量”。
- 从单进程观察扩展到子进程观察，引出**调度时序不确定性**（且有同一 boot 内 RUNNABLE/SLEEPING 混合的真实证据）。
- 增加两个负向测试（`pcount(99) = -1`、`pstate(-1) = -1`），让学生看到无效状态号与无效 pid 的返回约定。

该实验可以自然串联以下概念：

- `struct proc`、`enum procstate`、`proc[]`、`NPROC`
- `p->lock`：逐槽加锁、不可重入、非整表原子快照
- 调用进程在内核态不持有自身 `p->lock`（为何 `pcount` 不自死锁）
- syscall 参数传递（`argint`）与输入校验
- 子进程、`fork`/`kill`/`wait`、调度时序
- 内核观察结果的快照语义
- 用户态与内核态对状态编号的语义耦合

## 当前边界 / 不足

- 不是完整 `ps`。
- 不修改调度器。
- `pcount` 是逐槽加锁的近似聚合，不是整表原子快照。
- `pcount(RUNNING)` 数值随运行环境、`CPUS` 与时序变化，不固定承诺（实测 1）。
- `pchildtest` 的子进程状态可能为 `RUNNABLE`、`SLEEPING` 或其它有效状态，不固定承诺。
- timeout 自动捕获不是长期稳定性测试。
- `pchildtest`/`pcounttest` 依赖 baseline 的 `pause` 命名，仅在固定 baseline commit 下保证可编译。
