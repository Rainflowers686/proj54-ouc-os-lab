# 赛题要求与评分项拆解

## 信息来源说明

本文档根据队伍已记录的官方页面信息和当前项目理解整理，用于指导初赛 MVP 交付。若后续官方页面、通知或提交模板更新，应以官方最新要求为准，并在本文档中标注修订日期。

## 已知赛题信息

- 赛题：proj54 面向操作系统课程的操作系统竞赛和实验
- 赛题类型：教学型
- 难度等级：B
- 主要方向：
  - 往年操作系统比赛内核实现赛道的 step by step 指导教程。
  - 适合本校特点的实验指导教程，包括文档、代码框架、测例、参考实现。
- 评分要点：
  - 文档完备度：50%
  - 实现完整度：30%
  - 测试完整度：10%
  - 创新性：10%

## 赛题关键词拆解

| 关键词 | 理解 | 对项目的要求 |
| --- | --- | --- |
| 教学型 | 重点不是堆叠高难度功能，而是服务课程实验和学生学习 | 文档必须清楚，任务粒度适合低年级学生 |
| step by step | 过程比单个最终结果更重要 | 记录开发、fix bug、测试、复盘和 Git 阶段性成果 |
| 适合本校特点 | 需要贴合中国海洋大学低年级学生基础和课程节奏 | lab0 从环境配置开始，降低入门门槛 |
| 文档、代码框架、测例、参考实现 | 实验体系必须形成闭环 | 每个 lab 都规划教程、框架、测试和参考实现 |
| 引用和改造说明 | 可参考已有资料，但必须说明来源和改造 | 维护 references 与 license 文档 |
| 开源 | 完成代码和文档开源，已完成部分版权属于自己 | 自有文档和脚本清晰标注，第三方内容保留许可证 |
| 适合高校实验 | 最终评价目标是可教学、可复现、可扩展 | 不追求完整 OS，而追求可持续实验体系 |

## 选择“题目二为主、题目一为辅”的理由

根据已知赛题信息，题目一更偏向“往年操作系统比赛内核实现赛道的 step by step 指导教程”，题目二更偏向“适合本校特点的实验指导教程，包括文档、代码框架、测例、参考实现”。两者并非互斥，但**主攻方向决定资源分配与验收标准**，因此必须明确取舍。

### 两个方向的对比

| 维度 | 题目一（内核实现赛道 step by step 教程） | 题目二（适合本校的实验指导教程） |
| --- | --- | --- |
| 目标读者 | 已有较强基础、冲击内核实现赛道的选手 | 海大低年级、初次接触 OS 实验的学生 |
| 交付核心 | 复现高难度内核实现的全过程记录 | 文档 + 代码框架 + 测例 + 参考实现的完整实验体系 |
| 对队伍的要求 | 需要先有一份高质量内核实现作为讲解对象 | 需要把成熟 baseline（如 xv6-riscv）拆成可教学的渐进任务 |
| 与本队现状匹配度 | 低：本队尚无内核实现，27 天内难以同时完成实现与教程 | 高：可直接基于 xv6-riscv 设计 lab0-lab5 教学链路 |
| 风险 | 实现卡住则教程无对象，整体落空 | 即使深度实验受阻，仍可交付教学型成果（见止损线） |

### 选择题目二为主的理由

1. **读者匹配**：队伍目标用户是中国海洋大学低年级计算机学生，优先需要从环境、工具链、代码阅读和基础实验开始，这正是题目二的核心。
2. **可行性匹配**：题目一要求先有一份内核实现作为 step by step 的讲解对象，而本队当前无内核实现；在 27 天内同时完成「内核实现 + 实现教程」不现实。题目二只需把成熟 baseline 拆成教学任务，可行性显著更高。
3. **评价方向匹配**：题目二更容易体现“适合本校实验”的要求，与教学型赛题“可教学、可复现、可扩展”的最终评价方向一致。
4. **抗风险**：题目二即使深度实验受阻，仍能以「代码阅读 + 路径分析 + 实验设计」交付教学价值（详见 [00_project_plan.md](00_project_plan.md) 止损线），而题目一一旦实现卡住则整体落空。

### 题目一作为辅的落地方式

题目一的 step by step 过程记录价值不丢弃，而是降级为**贯穿全项目的方法论**：每个 lab 的教程结构、Git 阶段性提交、踩坑与修复记录（[03_step_by_step_guide.md](03_step_by_step_guide.md)、[06_progress_log.md](06_progress_log.md)、[07_faq_and_issues.md](07_faq_and_issues.md)）都体现 step by step 思想，但不单独冲击「内核实现赛道」这一高难度目标。

### 当前证据自评小结

- **已有证据**：文档完备度的「规划/治理/赛题拆解」子项、脚本可运行性、测试计划与模板。
- **暂无证据**：实现完整度（30%）与测试完整度（10%）几乎全部缺乏真实产物；文档完备度中最关键的「教学正文」尚未成形；创新性仍停留在主张层面。
- **结论**：当前最值得投入的不是再写更多规划文档，而是产出**真实可验证的 lab0+lab1 端到端证据**，以兑现实现、测试两项，并坐实文档与创新。详细判断见 [10_red_team_review.md](10_red_team_review.md)。

## 初赛 MVP 范围

初赛 MVP v0.1 聚焦可被评委理解的项目方案和可继续实施的实验结构：

- 完成正式 README 首页。
- 完成赛题要求与评分项拆解。
- 完成项目计划、lab0 环境教程、lab1 系统调用实验设计。
- 将脚本从纯 TODO 升级为不依赖 xv6 源码的最小可运行工具。
- 建立 AI 使用记录、进度日志、引用与许可证规范。
- 建立测试目录说明和初赛材料索引草案。

## 非目标范围

本轮明确不做以下事项：

- 不做完整 OS。
- 不做大型在线平台或自动评测系统。
- 不引入报名材料、个人隐私、账号密码、token、截图原件或大型二进制文件。
- 不引入 xv6-riscv 源码；仅为后续引入 baseline 做准备。
- 不伪造运行结果、测试通过记录、真实 commit hash、真实视频链接或性能数据。

## 风险分析与应对策略

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 范围过大 | 初赛阶段无法形成可复核成果 | 将 MVP 限定为文档、lab0/lab1 和脚本最小闭环，后续分阶段扩展 |
| 引用不规范 | 版权和开源合规风险 | 所有参考资料记录名称、URL、许可证、使用位置和改造说明 |
| AI 使用不透明 | 评审和队内复核难以判断贡献边界 | 在 docs/05_ai_usage_record.md 中记录使用场景、人工核验和影响文件 |
| 过程记录不足 | 难以体现 step by step 和 Git 阶段性成果 | 维护 docs/06_progress_log.md，并要求每 1-2 天至少一次有效 commit |
| 没有真实运行验证 | 文档可信度下降 | 未验证内容统一标注 TODO/待验证；引入 baseline 后补充真实命令和输出 |
| Windows/WSL 环境差异 | 低年级同学复现实验困难 | lab0 明确推荐环境、检查命令和常见问题，脚本保持宽容输出 WARN |
