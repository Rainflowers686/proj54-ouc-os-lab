# 项目计划

## 目标

本计划说明 OUC xv6 Lab Kit 的建设目标、交付边界和维护方法。项目面向操作系统课程实验，不追求大规模内核功能扩展，而是把 xv6-riscv 的系统调用、进程表、页表和文件表组织成可学习、可复现、可验收的实验链条。

## 适用对象

本文件服务于项目维护者、队长、助教和后续接手成员。学生不需要先读本计划，学生入口应是根 `README.md` 与 `labs/`。

## 内容范围

计划覆盖 Lab0 到 Lab5、独立 patch、integrated patch sequence、验证脚本、正式文档、证据材料和提交边界。当前正式版本采用 integrated suite `0001-0009`，historical `0001-0007` 只保留追溯价值。

## 工作结构

项目由五类产物构成：`labs/` 放课程学习路径；`patches/` 放可回放实现；`scripts/` 放环境、应用补丁、启动、测试和验收工具；`docs/` 放正式文档与历史审查；`submissions/` 放提交清单和证据索引。第三方 xv6 源码只在 `external/xv6-riscv/` 本地存在，不进入 Git。

## 里程碑

Lab0 解决环境和 baseline；Lab1 完成 `hello()` 与 `add2(int,int)`；Lab2 完成进程状态观察；Lab3 完成页表映射观察与 `memstat` 结构体 copyout；Lab4 完成 file/fd 观察与 `fdinfo`；Lab5 组织 clean baseline 到 full verification 的综合复现。

## 质量标准

每个阶段产物应能说明目标、命令、预期现象、失败处理和边界。补丁必须能从声明的 baseline 应用，验证必须来自真实 QEMU 输出，文档必须区分 当前正式验证范围 与 历史证据。

## 边界条件

不把教学观察接口夸大为完整内核子系统；不把队长本机验证写成队友独立复现；不把旧 suite 证据覆盖新 suite；不提交外部源码、日志原件、视频或隐私材料。

## 结构规范

文档应按“背景或问题、过程或设计、证据或命令、风险和后续动作”的顺序组织。历史文档可保留阶段性记录，但必须避免覆盖 当前正式验证范围。

## 语言风格

使用中文技术写作风格，命令、文件名、syscall 名和 SHA256 保持原样。结论应有证据支撑，不使用宣传性、绝对化或无法验证的表述。
